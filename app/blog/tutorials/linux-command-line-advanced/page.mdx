# Linux 与命令行：进阶与高效工作流

目标：在已经能正常使用服务器的基础上，进一步建立更稳定、更高效的命令行工作流。

继续阅读前，建议你先看完 [Linux 与命令行：入门必备](/blog/tutorials/linux-command-line)。

## 第二部分：进阶

前面的内容已经足够你正常登录服务器、整理文件、查看输出、做基础传输和完成最简单的批处理。

这一部分的目标不是“背更多命令”，而是开始建立更稳定的工作习惯，减少重复劳动，逐步把自己的环境和流程整理顺。

### 1. 权限和环境变量
#### 权限

为了各种保护，比如在一台集群上有很多用户，你总不希望别人可以随意查看篡改你的文件，这是一个很小的例子。因此各种系统都会有权限功能。
你可以先把权限理解成：系统要决定“谁可以对什么东西做什么事”。

放到文件上，最常见的就是三类权限：

- 读 `read`：能不能查看文件内容
- 写 `write`：能不能修改文件内容
- 执行 `execute`：能不能把它当成命令或程序直接运行

用 `ls -l` 可以看到一个文件的大致权限信息，例如：

```bash
ls -l run.sh # 查看 run.sh 的权限信息
```

输出通常会长得像这样：

```text
-rwxr-xr-- 1 user group 1234 Apr 6 10:00 run.sh
```

- 开头这一串字符是在描述权限
- `r` 表示可读（read）
- `w` 表示可写（write）
- `x` 表示可执行（execute）
- 如果某个位置是 `-`，就表示没有这一项权限

再往前走一步，Linux 里通常会把权限分给几类人：

- 文件所有者（user）：通常是创建这个文件的人
- 同组用户（group）：和你属于同一个用户组的人
- 其他用户（others）：既不是文件所有者，也不在同组里的其他人

像 `-rwxr-xr--` 这种写法，通常可以按三段看：

```text
- rwx r-x r--
```

也就是：

- 第一段 `rwx`：文件所有者的权限
- 第二段 `r-x`：同组用户的权限
- 第三段 `r--`：其他用户的权限

你还会经常看到另一种写法，比如：

```bash
chmod 755 run.sh   # 给脚本设置常见的可执行权限
chmod 644 data.txt # 给普通文本文件设置常见权限
```

这里：

- `755`：所有者可读可写可执行；同组和其他用户可读可执行
- `644`：所有者可读可写；同组和其他用户只读

更底层一点地看，这其实是把 `rwx` 三个位写成二进制，再转换成八进制数字：

```text
r = 100₂ = 4
w = 010₂ = 2
x = 001₂ = 1
```

所以每一位权限都是这三个二进制位的组合：

```text
rwx = 111₂ = 7
rw- = 110₂ = 6
r-x = 101₂ = 5
r-- = 100₂ = 4
```

三位数字分别对应：

- 第一位：文件所有者 `user`
- 第二位：同组用户 `group`
- 第三位：其他用户 `others`

所以：

```text
755 -> 7   5   5
     -> rwx r-x r-x

644 -> 6   4   4
     -> rw- r-- r--
```

这才是重点。不是去背 `755` 和 `644` 这两个例子，而是要能自己把任意一组数字权限拆出来。

比如你以后看到：

- `700`，就应该能读成 `rwx --- ---`
- `750`，就应该能读成 `rwx r-x ---`
- `600`，就应该能读成 `rw- --- ---`

对新手来说，最常遇到的问题不是“文件不能看”，而是“脚本不能执行”。

```bash
ls -l            # 查看文件权限
chmod +x run.sh  # 给脚本增加可执行权限
```

如果一个脚本没有执行权限，通常需要：

```bash
chmod +x run.sh # 给脚本增加可执行权限
./run.sh        # 直接执行脚本
```

#### 环境变量

```bash
echo $PATH                 # 查看当前 PATH 环境变量
which python               # 查看 python 命令实际来自哪里
which jdftx                # 查看 jdftx 命令实际来自哪里
export OMP_NUM_THREADS=4   # 给当前 shell 设置环境变量
```

这一节你至少要理解：

- `PATH` 决定了系统去哪找命令
- `which` 可以告诉你一个命令实际来自哪里
- `export` 可以给当前 shell 设置环境变量

### 2. 搜索、日志和报错排查

看输出文件时，最有效的方式通常不是从头读到尾，而是先搜关键词。

#### `grep`

```bash
grep "pattern" file      # 在文件中搜索指定字符串
grep -r "pattern" dir    # 递归搜索目录中的文件
grep -i error job.out    # 忽略大小写搜索 error
grep -i warning OUTCAR   # 忽略大小写搜索 warning
```

#### `tail -f`

```bash
tail -f job.out # 实时追踪日志新输出
```

这很适合实时看日志。

一个常见场景是：

```bash
grep -i error OUTCAR  # 先搜索报错关键词
tail -n 50 OUTCAR     # 再看最后 50 行上下文
```

先搜关键词，再看最后几十行，通常就能快速判断任务有没有出问题。

### 3. 进程和任务管理

当你怀疑程序卡住、重复运行、占满资源时，需要知道怎么查。

```bash
ps -ef | grep python # 查看和 python 相关的进程
top                  # 查看系统资源占用
kill 12345           # 终止进程号为 12345 的进程
jobs                 # 查看当前 shell 的后台任务
```

分别解决的问题：

- `ps -ef | grep xxx`：查某个程序是否在运行
- `top`：看系统资源占用
- `kill PID`：终止指定进程
- `jobs`：查看当前 shell 的后台任务

### 4. 退出服务器后，任务怎么继续跑：`screen` `nohup` `&` `tmux`

这一条很重要。

如果你在服务器上直接跑长任务，结果本地断网、终端关闭、SSH 断开，任务很可能也会跟着中断。

很多教程会先讲 `tmux`，但现实里不是所有服务器都装了 `tmux`。所以这里更实用的记法是：

- 有 `tmux` 或 `screen`：优先用会话管理工具
- 没有：至少会用 `nohup` 和 `&`

#### `screen`

有些服务器没有 `tmux`，但有 `screen`。它的作用和 `tmux` 很像，本质上都是“开一个不会因为 SSH 断开就立刻消失的终端会话”。

```bash
screen -S work     # 创建一个名为 work 的 screen 会话
python train.py    # 在会话里运行任务
```

想暂时离开会话时，按：

```text
Ctrl-a d
```

之后可以重新接回去：

```bash
screen -ls         # 查看当前已有的 screen 会话
screen -r work     # 重新连接名为 work 的会话
```

#### `tmux`

如果服务器装了 `tmux`，那它通常是更舒服的选择。它和 `screen` 的核心用途一样，都是帮你保住一个可恢复的会话。

```bash
tmux new -s work   # 创建一个名为 work 的 tmux 会话
python train.py    # 在 tmux 会话里运行任务
```

想暂时离开时，按：

```text
Ctrl-b d
```

之后可以重新接回去：

```bash
tmux ls            # 查看当前已有的 tmux 会话
tmux attach -t work # 重新连接名为 work 的会话
```

#### `nohup` 和 `&`

如果服务器上既没有 `tmux`，也没有 `screen`，那最少要会这一种。

```bash
nohup python train.py > run.log 2>&1 & # 让任务在后台运行，并把输出写入日志
```

这里分开看：

- `nohup`：即使你退出终端，程序也尽量继续运行
- `> run.log`：把标准输出写入日志
- `2>&1`：把报错信息也写到同一个日志里
- `&`：让命令在后台运行

启动后可以继续检查：

```bash
tail -f run.log              # 实时查看日志
ps -ef | grep train.py       # 查看进程是否还在运行
jobs                         # 查看当前 shell 的后台任务
```

要注意一件事：`jobs` 只能看到当前 shell 启动的后台任务。你重新登录服务器之后，通常还是要靠 `ps` 去找进程。

#### 什么时候用哪一种

- `screen` / `tmux`：适合你还想随时回到原来的交互环境，继续看输出、改命令、重新运行
- `nohup ... &`：适合“一条命令直接挂后台”的场景

不过这里要补一句更重要的现实情况：

很多组里的正式计算并不是直接在登录节点里裸跑，而是通过集群队列系统提交任务。常见的有：

- `LSF`
- `Slurm`（常见命令是 `sbatch`）
- `PBS`

如果你的服务器有这些队列系统，那正式任务一般应该优先走队列，而不是长期挂在登录节点里用 `screen`、`tmux` 或 `nohup` 跑。后面学会提交脚本之后，你会发现这才是更规范的工作方式。

如果只记一句话，可以记这个：

> 长任务不要裸跑。临时交互任务可以用 `screen` / `tmux` / `nohup`，正式计算优先走队列系统。

### 5. 重定向和管道：`>` `>>` `|`

这是命令行真正开始“组合起来工作”的地方。

```bash
python script.py > output.log  # 覆盖写入日志
python script.py >> output.log # 追加写入日志
```

- `>`：覆盖写入
- `>>`：追加写入

管道示例：

```bash
tail -n 100 job.out | grep energy # 查看最后 100 行并筛选 energy
cat job.out | grep error          # 输出文件内容并筛选 error
```

虽然很多时候可以直接写成：

```bash
grep error job.out # 直接在文件里搜索 error
```

但理解管道依然很重要，因为后面很多工作流都是把小命令串起来完成的。

### 6. 最常见的压缩与解压：`tar`

这也是高频操作。结果文件很多的时候，经常需要打包、传输和备份。

```bash
tar -czvf dir.tar.gz dir # 将目录压缩成 tar.gz
tar -xzvf dir.tar.gz     # 解压 tar.gz 文件
```

如果你现在完全是新手，先把这两条记住就够了。

### 7. 批量查找与处理文件：`find` `xargs` 通配符

当目录结构一复杂，`find` 就会变得非常有用。

```bash
find . -name "*.out"         # 查找当前目录下所有 out 文件
find . -type d -name "calc*" # 查找名字以 calc 开头的目录
```

通配符也很常用：

```bash
ls *.vasp              # 列出当前目录下所有 vasp 文件
cp */CONTCAR summary/  # 复制各子目录中的 CONTCAR 到 summary 目录
```

如果要把查找到的文件继续交给别的命令处理，就可以配合 `xargs`：

```bash
find . -name "*.out" | xargs grep error # 查找所有 out 文件并搜索 error
```

### 8. 文本处理：`awk` `sed`

这两个命令是提取结果和批量替换参数的利器。

#### `awk`

```bash
awk '{print $1}' file      # 输出文件的第一列
awk -F: '{print $1}' file  # 以冒号为分隔符输出第一列
```

#### `sed`

```bash
sed 's/old/new/g' file # 将文件中的 old 替换成 new
```

你现在不需要把它们学得很深，但至少要知道：

- `awk` 很适合取列
- `sed` 很适合替换文本

很多结果提取脚本，其实就是这两个命令在干活。

### 9. 批处理脚本：`for`

Shell 真正强大的地方，是它很适合做批处理。

```bash
for i in {01..10}; do
  echo $i  # 逐个输出 01 到 10
done       # 循环结束
```

在科研里更常见的是遍历文件或目录：

```bash
for i in *; do
  cp $i/CONTCAR zzz/$i.vasp # 复制每个目录下的 CONTCAR 并重命名
done                        # 循环结束
```

```bash
for i in *.vasp; do
  mkdir ${i/.vasp}          # 用文件名创建同名目录
  mv $i ${i/.vasp}/POSCAR   # 把文件移动进去并改名为 POSCAR
done                        # 循环结束
```

```bash
for i in *; do
  echo -n "$i "                          # 先输出目录名，不换行
  tail -n 1 $i/331/OSZICAR | awk '{print $5}' # 提取最后一行的第 5 列
done                                     # 循环结束
```

这里更重要的不是背语法，而是慢慢建立感觉：

- 重复劳动通常都能写成循环
- 目录结构一旦统一，就非常容易自动化

### 10. 终端环境优化：`.zshrc` `alias` `oh-my-zsh` `fzf`

当你已经开始高频使用命令行时，可以逐步优化自己的环境。

最基础的是 `alias`：

```bash
alias ll='ls -lh'    # 给 ls -lh 起一个简短别名
alias gs='git status' # 给 git status 起一个简短别名
```

如果你使用 `zsh`，可以进一步配置：

- `oh-my-zsh`
- `zsh-autosuggestions`
- `zsh-syntax-highlighting`
- `fzf`

这些工具的意义不是“好看”，而是减少输入量、降低出错率、提高历史命令和文件搜索效率。

如果你已经开始高频使用服务器，也可以继续配这些：

#### `oh-my-zsh`

如果服务器已经安装了 `zsh`，可以再装 `oh-my-zsh`：

```bash
sh -c "$(curl -fsSL https://raw.githubusercontent.com/ohmyzsh/ohmyzsh/master/tools/install.sh)" # 安装 oh-my-zsh
```

它的配置文件是 `~/.zshrc`。

#### 自动补全和语法高亮

常用插件：

- `zsh-autosuggestions`
- `zsh-syntax-highlighting`

安装示例：

```bash
git clone https://github.com/zsh-users/zsh-autosuggestions ${ZSH_CUSTOM:-~/.oh-my-zsh/custom}/plugins/zsh-autosuggestions # 安装自动补全插件
git clone https://github.com/zsh-users/zsh-syntax-highlighting.git ${ZSH_CUSTOM:-~/.oh-my-zsh/custom}/plugins/zsh-syntax-highlighting # 安装语法高亮插件
```

然后在 `~/.zshrc` 的 `plugins=(...)` 里加上对应插件名。

#### `fzf`

`fzf` 是命令行模糊搜索工具，可以帮你更快找文件、找历史命令。

```bash
conda install -c conda-forge fzf # 安装 fzf 模糊搜索工具
```

配置思路通常是：

```bash
plugins=(git extract zsh-autosuggestions zsh-syntax-highlighting fzf) # 启用常用插件
FZF_BASE=/your/env/path/share/fzf/shell # 指定 fzf 安装路径
```

保存后重新登录或重新加载配置。

#### `extract`

这是一个解压相关的小工具插件。不是必须，但对于经常处理压缩文件的人很省事。

但我的建议还是一样：先掌握基本命令，再做环境优化。不要本末倒置。

### 11. 磁盘空间查看：`du` `df`

磁盘空间问题在服务器上很常见，尤其是结果目录越来越大之后。

```bash
du -sh .  # 查看当前目录总大小
du -sh *  # 查看各子目录大小
df -h     # 查看磁盘空间使用情况
```

这些命令分别用来：

- 看当前目录一共占多大
- 看各个子目录分别占多大
- 看磁盘是不是快满了

### 12. 压缩、解压和并行压缩：`tar` `pigz`

结果文件多了以后，压缩和备份是日常操作。

```bash
tar -czvf results.tar.gz results/ # 压缩 results 目录
tar -xzvf results.tar.gz          # 解压 results.tar.gz
```

如果装了 `pigz`，可以做并行压缩：

```bash
tar -cf - results | pigz > results.tar.gz # 使用 pigz 并行压缩目录
pigz -d -c results.tar.gz | tar -xf -     # 解压 pigz 压缩结果
```

这在大目录打包时会比普通 `gzip` 更快。

如果你经常需要压缩大目录，可以直接准备一个简化脚本，比如 `pgtar`：

```bash
#!/bin/zsh
tar -cf - $1 | pigz > $1.tar.gz # 并行压缩传入目录
```

解压对应写法：

```bash
pigz -d -c *.tar.gz | tar -xf - # 解压当前目录下所有 tar.gz 文件
```

### 13. 软链接：`ln -s`

软链接很像 Windows 里的快捷方式。

```bash
ln -s /home/user/file.txt /home/user/links/mylink # 创建软链接
```

它很适合：

- 简化很长的路径
- 给常用目录建入口
- 不复制文件本体，只建立引用

## 进阶检查清单

如果你已经把前面的进阶内容过了一遍，至少可以试着做到下面这些事：

1. 看懂 `ls -l` 里的权限信息，并知道 `chmod +x`、`chmod 755`、`chmod 644` 的基本用途
2. 用 `grep`、`tail -f`、`ps` 排查一个正在运行的任务
3. 知道什么时候该用 `screen` / `tmux`，什么时候该用 `nohup ... &`

6. 给自己的终端加一个 `alias`，或者简单配置一次 `~/.zshrc`

## 常见坑

### 1. 分不清本地和服务器

很多“文件不见了”“路径不对”的问题，其实只是你搞混了自己现在到底在本地还是服务器。

### 2. 不看当前目录就执行删除命令

这个坑非常危险。删之前先：

```bash
pwd # 先确认当前目录
ls  # 再确认当前目录下的文件
```

### 3. 直接把长任务裸跑在当前终端里

这是新人很值得尽早避开的一个坑。

一旦本地断网、终端关闭，或者 SSH 连接断开，任务就可能一起中断。

更稳的做法是：

- 临时交互任务用 `screen` / `tmux`
- 简单后台命令用 `nohup ... &`
- 正式计算优先走队列系统

### 4. 用 `cat` 打开超大文件

大文件优先用：

```bash
less file         # 翻页查看大文件
tail -n 50 file   # 看最后 50 行
grep keyword file # 搜索关键词
```

### 5. 不理解 `>` 和 `>>` 的区别

一个是覆盖，一个是追加。这个没搞清楚，日志很容易被你自己覆盖掉。

## 最后一段

对新人来说，命令行最重要的不是“会多少命令”，而是先建立一套最基本的工作能力：

- 连上服务器
- 找到文件
- 看懂日志
- 保住长任务
- 批量处理重复工作

做到这一步，命令行就已经开始真正帮你节省时间了。
