# Linux 与命令行

目标：帮助你尽快具备独立使用命令行控制 Linux 服务器的能力。

## 前言

### 为什么学习命令行？

1. Linux 服务器主要就是通过命令行使用的。
2. 所有的科学计算软件 (VASP, LAMMPS...) 都需要通过命令行实现。

### CLI 和 GUI

说到电脑，很多同学更熟悉的是图形界面（Graphical User Interface, GUI），比如点开文件夹、双击运行程序、通过窗口切换任务。
但在科研计算环境里，命令行（Command Line Interface, CLI）通常更重要。区别如下：

| 交互方式 | 典型操作 | 优点 | 局限 |
| --- | --- | --- | --- |
| GUI | 鼠标点点点，在对应的地方输入对应的东西 | 直观、上手快 | 不适合远程、大批量、自动化任务 |
| CLI | 在终端输入文本命令 | 逻辑简单，高效 | 不够直观，初学时有门槛 |

对科学计算来说，CLI 更重要，不是因为它“看起来更专业”，而是因为它更适合下面这些场景：

- 远程操作服务器
- 批量处理文件
- 自动化重复任务
- 记录和复现完整流程


### 基础和进阶
我把内容分成两部分：

- **基础**：这是入门的最小闭环，后面基本都会用到。
- **进阶**：不是第一天必须全会，但建议先知道有这些工具。很多时候效率上不去，不是因为命令难，而是不知道 shell 原来可以这样做。

如果你完全没用过 Linux，我建议你先把“基础”过一遍，并亲自操作。不要试图一口气背下来，重点是知道这些命令分别在干什么。

## 第一部分：基础

### 1. 登录服务器：`ssh`

第一件事是登录服务器。

Windows 用户可以用 PowerShell，或者更推荐直接装 `WSL`。如果你更习惯图形化终端工具，也可以先用 [`Xshell`](https://www.netsarang.com/en/xshell/) 、[`MobaXterm`](https://mobaxterm.mobatek.net/) 这类软件连接服务器。macOS 用户直接打开终端即可。

#### 密码连接

```bash
ssh -p port username@hostname # 用指定端口连接远程服务器

# -p 后面是端口号，如果端口是默认值 22 可以不写
# username 是用户名
# hostname 一般是 IP 或者域名
```

回车执行命令后，一般是让你输入密码，输入即可登录成功。

注意：终端内的密码不会明文显示，即输入是没反应的，不是卡住了。

#### 密钥连接

如果你经常登录同一台服务器，建议配置 `~/.ssh/config` 并且使用密钥连接：

```ssh-config
Host alias                       # 给这台服务器起一个本地别名
    HostName hostname            # 服务器地址，通常是 IP 或域名
    User username                # 登录用户名
    Port port                    # SSH 端口
    IdentityFile path/to/private/key # 私钥路径
```

这样以后只要：

```bash
ssh alias # 使用别名连接服务器
```

这样做的好处在于以后：

- `ssh`
- `scp`
- `rsync`
- `sftp`
- `...`

这些和服务器通信的命令也都可以直接复用这个别名，不用每次重新记 `IP`、用户名和端口。

#### 密钥配置

刚刚解决了每次重复输入`IP`、用户名和端口的问题，我们现在解决每次都要输入密码的问题。

#### 使用 `ssh-keygen` 生成密钥对

```bash
ssh-keygen # 生成一对 SSH 密钥
# 一直按回车就好了，感兴趣的可以看看在干嘛
```

终端会提示你保存路径和密码。默认一般是：

```text
~/.ssh/id_rsa
~/.ssh/id_rsa.pub
```

其中：

- `id_rsa` 是私钥
- `id_rsa.pub` 是公钥

`IdentityFile` 对应的就是私钥路径。默认情况下通常就是 `~/.ssh/id_rsa`。

#### 使用 `ssh-copy-id` 复制公钥到服务器

如果本地有 `ssh-copy-id`，这是最省事的做法：

```bash
ssh-copy-id alias # 将本地公钥复制到服务器 authorized_keys
```

它会把你的公钥自动追加到服务器的 `~/.ssh/authorized_keys` 中。

#### 手动复制公钥

如果没有 `ssh-copy-id`，也可以手动复制 `id_rsa.pub` 里的内容，粘贴到服务器的：

```text
~/.ssh/authorized_keys
```

如果这个文件里已经有内容，就另起一行追加，不要覆盖原来的内容。

配置完成后，`ssh alias` 一般就可以直接通过密钥连接，不用每次重新输入密码。

这一节最少要做到：

- 能独立登录服务器
- 知道本地终端和远程终端不是一回事
- 知道 `config` 文件能显著减少重复输入
- 知道私钥和公钥分别是什么
- 知道 `authorized_keys` 的作用

### 2. 文件和目录操作

这些命令是日常频率最高的一组。这里后续只给出简单的例子，如果感兴趣可以问 LLM Model，应该会更有收获。

这里是文件和目录操作最经典的一些命令：

| 命令 | 全称 | 主要作用 |
| --- | --- | --- |
| `pwd` | print working directory | 显示当前所在目录 |
| `ls` | list | 列出当前目录下的文件和文件夹 |
| `cd` | change directory | 切换目录 |
| `mkdir` | make directory | 创建目录 |
| `cp` | copy | 复制文件或目录 |
| `mv` | move | 移动文件，或者给文件重命名 |
| `rm` | remove | 删除文件或目录 |

可以看到其实就是英文的首字母缩写，然后通过添加参数实现各种功能。

最常见的用法：

```bash
pwd                        # 显示当前所在目录
ls                         # 列出当前目录下的文件和文件夹
ls -l                      # 以详细格式显示文件信息
ls -a                      # 显示隐藏文件
ls -t                      # 按修改时间排序，新的排前面
ls -r                      # 反序显示
ls -R                      # 递归显示子目录内容

cd ..                      # 切换到上一级目录
cd -                       # 切换回上一次所在的目录
cd ~                       # 切换到用户家目录
cd /                       # 回到根目录
cd relative/path           # 进入相对路径
cd /absolute/path          # 进入绝对路径

mkdir test                 # 创建一个名为 test 的目录
mkdir -p project/run1/logs # 递归创建多级目录

cp input.in backup.in      # 复制文件并另存为 backup.in
cp -r old_dir new_dir      # 递归复制整个目录

mv file1 file2             # 将 file1 重命名为 file2
mv *.txt results/          # 将当前目录下所有 txt 文件移动到 results 目录
```

其中 `ls -t` 很有用。文件多的时候，你很容易忘记最近动过哪个目录，按修改时间排序通常能快速定位。

#### `Tab` 补全

这也是非常值得尽早养成的习惯。

在终端里输入一部分文件名、目录名或者命令后，按 `Tab` 键，shell 会尝试帮你自动补全。

比如：

```bash
cd proj<Tab>          # 自动补全目录名
cp INC<Tab> backup    # 自动补全文件名
ssh gp<Tab>           # 自动补全别名或命令
```

它的好处很直接：

- 少打字
- 少拼写错误
- 文件路径很长时更省事

但也要注意：`Tab` 补全虽然方便，不代表它补出来的就一定是你真正想操作的目标。尤其是在执行 `rm`、`mv` 这类命令时，补全之后还是建议先看一眼，避免选错对象。

`rm` (remove) 是最需要小心的命令之一。

```bash
rm file    # 删除一个文件
rm -r dir  # 递归删除目录
rm -rf dir # 强制递归删除目录，危险
```
`rm` 没有回收站没有废纸篓，删了就真没了，几乎不可能恢复，就算能也非常费劲。

我想几乎所有人都会遇上误删的经历。所以这里给一个很实际的建议：

- 尽量不要形成无脑 `rm -rf` 的习惯
- 删除前先 `pwd`
- 再 `ls`
- 再确认当前目录和目标路径

一个更稳妥的删除目录思路是：

1. 先 `cd` 进入目标目录
2. 用 `rm -r *` 删除内部内容
3. 确认目录已空
4. 回到上一级后再 `rmdir`

这比直接在外面敲一条大而危险的 `rm -rf some_dir` 要稳很多。

### 3. 查看文件内容：`cat` `less` `head` `tail`

做计算科研时，你每天都在看输入文件和输出文件。

| 命令 | 全称 | 主要作用 |
| --- | --- | --- |
| `cat` | concatenate | 直接输出文件内容，适合小文件 |
| `less` | less | 分页查看文件，适合大文件 |
| `head` | head | 查看文件开头几行 |
| `tail` | tail | 查看文件最后几行，或持续追踪新输出 |

例如：

```bash
cat INCAR          # 直接输出小文件内容
head -n 20 OUTCAR  # 查看前 20 行
tail -n 20 OUTCAR  # 查看后 20 行
less OUTCAR        # 翻页查看大文件
```

简单理解：

- 小文件用 `cat`
- 大文件优先用 `less`
- 看文件开头用 `head`
- 看文件结尾和最新输出用 `tail`

还有一个很实用的用法：

```bash
tail -n 1 */result # 查看多个 result 文件的最后一行
```

它可以直接查看当前目录下一批结果文件的最后一行，在批量计算里很常见。

### 4. 文件传输：`scp` 和 `rsync`

本地电脑和服务器是不能直接互相读取的对方的文件的，所以有时候需要把计算任务上传到服务器，然后把计算完的数据下载到本地做数据分析处理。

#### `scp`

```bash
# 从本地传文件到服务器
scp local.txt alias:~/work/        # 上传单个文件

# 从服务器传文件到本地
scp alias:~/work/result.txt ./     # 下载单个文件到当前目录

# 传目录时加 -r
scp -r local_dir alias:~/work/     # 上传整个目录
```

观察这个命令的结构其实很重要：

```bash
scp <source> <destination> # 从源路径复制到目标路径
```

这和本地复制命令 `cp` 很像，只是 `scp` 可以跨机器传输。

#### `rsync`

```bash
rsync -avh local_dir/ alias:~/work/local_dir/ # 本地同步目录到服务器
rsync -avh alias:~/work/results/ ./results/   # 服务器同步目录到本地
```

和 `scp` 相比，`rsync` 更像“同步”，不是单纯复制。

常见用法：

```bash
# 本地同步到服务器
rsync -avh local_dir/ alias:~/work/local_dir/ # 同步整个目录到服务器

# 服务器同步到本地
rsync -avh alias:~/work/results/ ./results/   # 同步整个目录到本地

# 只同步某类文件
rsync -avh *.txt alias:~/work/                # 同步当前目录下所有 txt 文件
```

其中常见参数：

- `-a`：归档模式，尽量保留文件信息
- `-v`：显示过程
- `-h`：人类可读的文件大小

`rsync` 有一个很重要的特点：它会比较源和目标之间的差异，所以非常适合反复同步同一个目录。

这意味着如果你经常做这些事情：

- 本地改脚本，再同步到服务器
- 服务器跑完结果，再同步回本地
- 中途中断后重新传输

那 `rsync` 往往会比 `scp` 更省时间，也更稳。

简单区分：

- 临时传一个文件，用 `scp`
- 需要同步很多文件、重复同步、希望更稳一点，优先用 `rsync`

#### 一个很实际的问题

`scp` 虽然不难，但实际使用起来：你在本地终端里，不一定记得服务器上的完整路径；你在服务器终端里，也不方便直接知道本地路径。你每次都得输入一串很长的目录名，非常麻烦。能不能偷懒？
能当然能！

我自己使用方法则是在服务器上准备一个 `put`：

```bash
#!/bin/bash
echo "scp -r alias:$PWD/$1 ." | pbcopy # 生成下载命令并复制到本地剪贴板
```

如果你是第一次看到这种脚本格式，可以先这样理解：

- `#!/bin/bash`：表示这个脚本用 `bash` 来执行
- `echo`：把一段文本输出出来
- `|`：管道，表示把前一个命令的输出交给后一个命令

所以这一行命令本质上是在做一件事：先生成一条 `scp` 命令，再把这条命令交给 `pbcopy`，复制到剪贴板里。

这个脚本做的事很简单：

- 读取你当前服务器目录 `PWD`
- 拼出一条下载命令
- 复制到本地剪贴板

然后你回到本地终端，直接粘贴执行就行。

这里的 `pbcopy` 是 macOS 自带的剪贴板命令。如果你本地是 macOS，这种用法会很顺手。记得打开终端工具允许写入剪贴板的功能。

如果你使用的是 `iTerm2`，或者其他支持 `OSC 52` 的终端，也可以直接用下面这种写法：

```bash
#!/bin/bash
CMD="scp -r alias:$PWD/$1 ."                         # 先生成要执行的 scp 命令
ENCODED=$(printf "%s" "$CMD" | base64 | tr -d '\n') # 将命令做 base64 编码并去掉换行
printf '\e]52;c;%s\a' "$ENCODED"                     # 通过 OSC 52 协议把内容发送到本地剪贴板
```

这里多解释一句：

- `CMD=...`：先把整条命令存到一个变量里
- `ENCODED=...`：把这条命令编码成终端可安全传输的格式
- `printf '\e]52;c;%s\a' ...`：通过 `OSC 52` 协议请求终端写入本地剪贴板

这个方法的好处是不依赖服务器上有没有 `pbcopy`。只要你的本地终端支持 `OSC 52`，就可以直接把远程输出送回本地剪贴板。

如果服务器端没有 `pbcopy`，也可以用一个简单的 Python 脚本实现类似效果：

```python
#!/usr/bin/env python
import sys      # 读取标准输入
import base64   # 把文本编码成终端可识别的剪贴板格式

buffer = []     # 保存输入内容
for line in sys.stdin:
    buffer.append(line)  # 逐行收集输入

content = ''.join(buffer).strip()  # 拼成完整字符串并去掉首尾空白
print('\033]52;c;' + base64.b64encode(content.encode()).decode() + '\007')  # 发送到本地剪贴板
```

这个思路的重点不在于“必须这样做”，而在于你要开始意识到：

- 命令行不只是手动敲命令
- 很多高频动作都可以给自己做一层小封装

同理，在本地你也可以准备一个简单脚本，用来把文件上传到服务器：

```bash
#!/bin/zsh
scp -r $1 alias:~ # 将本地文件或目录上传到服务器家目录
```

先把文件传到服务器家目录，再在服务器里用 `mv` 挪到目标位置。这个方式不一定是最优雅的，但对日常使用很实用，也足够稳。

最后补充一句：文件传输当然也有很多图形化界面工具，比如一些 SSH 客户端自带的文件面板、SFTP 文件浏览器之类。常见的 Windows 工具包括 [Xshell](https://www.netsarang.com/en/xshell/) 和 [MobaXterm](https://mobaxterm.mobatek.net/)。它们用起来更直观，但底层本质上还是这些远程传输协议和命令的封装。

另外还有一些在服务器环境里常见的工具，比如 [`lrzsz`](https://ohse.de/uwe/software/lrzsz.html)（`rz` / `sz`），也可以做文件上传下载。这里只先知道有这些东西就够了，不展开讲。感兴趣的话可以自己再查。

### 5. 权限和环境变量：`chmod` `which` `echo $PATH` `export`

很多“命令不能执行”的问题，本质上不是程序本身有 bug，而是权限或环境变量没配对。

#### 权限

```bash
ls -l            # 查看文件权限
chmod +x run.sh  # 给脚本增加可执行权限
```

如果一个脚本没有执行权限，通常需要：

```bash
chmod +x run.sh # 给脚本增加可执行权限
./run.sh        # 直接执行脚本
```

#### 环境变量

```bash
echo $PATH                 # 查看当前 PATH 环境变量
which python               # 查看 python 命令实际来自哪里
which jdftx                # 查看 jdftx 命令实际来自哪里
export OMP_NUM_THREADS=4   # 给当前 shell 设置环境变量
```

这一节你至少要理解：

- `PATH` 决定了系统去哪找命令
- `which` 可以告诉你一个命令实际来自哪里
- `export` 可以给当前 shell 设置环境变量

### 6. 搜索、日志和报错排查：`grep` `tail -f`

看输出文件时，最有效的方式通常不是从头读到尾，而是先搜关键词。

#### `grep`

```bash
grep "pattern" file      # 在文件中搜索指定字符串
grep -r "pattern" dir    # 递归搜索目录中的文件
grep -i error job.out    # 忽略大小写搜索 error
grep -i warning OUTCAR   # 忽略大小写搜索 warning
```

#### `tail -f`

```bash
tail -f job.out # 实时追踪日志新输出
```

这很适合实时看日志。

一个常见场景是：

```bash
grep -i error OUTCAR  # 先搜索报错关键词
tail -n 50 OUTCAR     # 再看最后 50 行上下文
```

先搜关键词，再看最后几十行，通常就能快速判断任务有没有出问题。

### 7. 进程和任务管理：`ps` `top` `kill` `jobs`

当你怀疑程序卡住、重复运行、占满资源时，需要知道怎么查。

```bash
ps -ef | grep python # 查看和 python 相关的进程
top                  # 查看系统资源占用
kill 12345           # 终止进程号为 12345 的进程
jobs                 # 查看当前 shell 的后台任务
```

分别解决的问题：

- `ps -ef | grep xxx`：查某个程序是否在运行
- `top`：看系统资源占用
- `kill PID`：终止指定进程
- `jobs`：查看当前 shell 的后台任务

### 8. 长任务保活：`tmux`

这一条非常重要。

如果你在服务器上直接跑长任务，结果本地断网、终端关闭、SSH 断开，任务很可能也跟着断掉。

最简单有效的解决方案是 `tmux`。

```bash
tmux new -s work # 创建一个名为 work 的 tmux 会话
```

进入会话后运行任务。要暂时离开时，按：

```text
Ctrl-b d
```

重新连接：

```bash
tmux attach -t work # 重新连接名为 work 的会话
```

查看已有会话：

```bash
tmux ls # 查看当前已有的 tmux 会话
```

一句话记住：

> 只要任务可能跑很久，就先开 `tmux`。

### 9. 重定向和管道：`>` `>>` `|`

这是命令行真正开始“组合起来工作”的地方。

```bash
python script.py > output.log  # 覆盖写入日志
python script.py >> output.log # 追加写入日志
```

- `>`：覆盖写入
- `>>`：追加写入

管道示例：

```bash
tail -n 100 job.out | grep energy # 查看最后 100 行并筛选 energy
cat job.out | grep error          # 输出文件内容并筛选 error
```

虽然很多时候可以直接写成：

```bash
grep error job.out # 直接在文件里搜索 error
```

但理解管道依然很重要，因为后面很多工作流都是把小命令串起来完成的。

### 10. 最常见的压缩与解压：`tar`

这也是高频操作。结果文件很多的时候，经常需要打包、传输和备份。

```bash
tar -czvf dir.tar.gz dir # 将目录压缩成 tar.gz
tar -xzvf dir.tar.gz     # 解压 tar.gz 文件
```

如果你现在完全是新手，先把这两条记住就够了。

---

## 第二部分：进阶掌握

这一部分不是第一天必须全会，但建议你尽早知道有这些工具。真正拉开效率差距的，往往不是“会不会 `ls`”，而是会不会把重复工作慢慢自动化。

### 1. 批量查找与处理文件：`find` `xargs` 通配符

当目录结构一复杂，`find` 就会变得非常有用。

```bash
find . -name "*.out"         # 查找当前目录下所有 out 文件
find . -type d -name "calc*" # 查找名字以 calc 开头的目录
```

通配符也很常用：

```bash
ls *.vasp              # 列出当前目录下所有 vasp 文件
cp */CONTCAR summary/  # 复制各子目录中的 CONTCAR 到 summary 目录
```

如果要把查找到的文件继续交给别的命令处理，就可以配合 `xargs`：

```bash
find . -name "*.out" | xargs grep error # 查找所有 out 文件并搜索 error
```

### 2. 文本处理：`awk` `sed`

这两个命令是提取结果和批量替换参数的利器。

#### `awk`

```bash
awk '{print $1}' file      # 输出文件的第一列
awk -F: '{print $1}' file  # 以冒号为分隔符输出第一列
```

#### `sed`

```bash
sed 's/old/new/g' file # 将文件中的 old 替换成 new
```

你现在不需要把它们学得很深，但至少要知道：

- `awk` 很适合取列
- `sed` 很适合替换文本

很多结果提取脚本，其实就是这两个命令在干活。

### 3. 批处理脚本：`for`

Shell 真正强大的地方，是它很适合做批处理。

```bash
for i in {01..10}; do
  echo $i  # 逐个输出 01 到 10
done       # 循环结束
```

在科研里更常见的是遍历文件或目录：

```bash
for i in *; do
  cp $i/CONTCAR zzz/$i.vasp # 复制每个目录下的 CONTCAR 并重命名
done                        # 循环结束
```

```bash
for i in *.vasp; do
  mkdir ${i/.vasp}          # 用文件名创建同名目录
  mv $i ${i/.vasp}/POSCAR   # 把文件移动进去并改名为 POSCAR
done                        # 循环结束
```

```bash
for i in *; do
  echo -n "$i "                          # 先输出目录名，不换行
  tail -n 1 $i/331/OSZICAR | awk '{print $5}' # 提取最后一行的第 5 列
done                                     # 循环结束
```

这里更重要的不是背语法，而是慢慢建立感觉：

- 重复劳动通常都能写成循环
- 目录结构一旦统一，就非常容易自动化

### 4. 终端环境优化：`.zshrc` `alias` `oh-my-zsh` `fzf`

当你已经开始高频使用命令行时，可以逐步优化自己的环境。

最基础的是 `alias`：

```bash
alias ll='ls -lh'    # 给 ls -lh 起一个简短别名
alias gs='git status' # 给 git status 起一个简短别名
```

如果你使用 `zsh`，可以进一步配置：

- `oh-my-zsh`
- `zsh-autosuggestions`
- `zsh-syntax-highlighting`
- `fzf`

这些工具的意义不是“好看”，而是减少输入量、降低出错率、提高历史命令和文件搜索效率。

但我的建议是：先掌握基本命令，再做环境美化。不要本末倒置。

### 5. 后台任务和资源查看：`nohup` `du` `df`

有些任务不一定开交互式会话，也可能直接后台跑。

```bash
nohup python script.py > run.log 2>&1 & # 后台运行脚本并把输出写入日志
```

此外，磁盘空间问题也很常见：

```bash
du -sh .  # 查看当前目录总大小
du -sh *  # 查看各子目录大小
df -h     # 查看磁盘空间使用情况
```

这些命令分别用来：

- 后台运行任务
- 看目录大小
- 看子目录各占多少空间
- 看磁盘是否快满了

### 6. 压缩、解压和并行压缩：`tar` `pigz`

结果文件多了以后，压缩和备份是日常操作。

```bash
tar -czvf results.tar.gz results/ # 压缩 results 目录
tar -xzvf results.tar.gz          # 解压 results.tar.gz
```

如果装了 `pigz`，可以做并行压缩：

```bash
tar -cf - results | pigz > results.tar.gz # 使用 pigz 并行压缩目录
pigz -d -c results.tar.gz | tar -xf -     # 解压 pigz 压缩结果
```

这在大目录打包时会比普通 `gzip` 更快。

如果你经常需要压缩大目录，可以直接准备一个简化脚本，比如 `pgtar`：

```bash
#!/bin/zsh
tar -cf - $1 | pigz > $1.tar.gz # 并行压缩传入目录
```

解压对应写法：

```bash
pigz -d -c *.tar.gz | tar -xf - # 解压当前目录下所有 tar.gz 文件
```

### 7. 软链接：`ln -s`

软链接很像 Windows 里的快捷方式。

```bash
ln -s /home/user/file.txt /home/user/links/mylink # 创建软链接
```

它很适合：

- 简化很长的路径
- 给常用目录建入口
- 不复制文件本体，只建立引用

### 8. 终端环境配置：`.zshrc` `oh-my-zsh` `fzf`

这部分不是第一天必须做，但如果你已经开始高频用服务器，值得找个时间尽早配一下。

#### `oh-my-zsh`

如果服务器已经安装了 `zsh`，可以再装 `oh-my-zsh`：

```bash
sh -c "$(curl -fsSL https://raw.githubusercontent.com/ohmyzsh/ohmyzsh/master/tools/install.sh)" # 安装 oh-my-zsh
```

它的配置文件是 `~/.zshrc`。

#### 自动补全和语法高亮

常用插件：

- `zsh-autosuggestions`
- `zsh-syntax-highlighting`

安装示例：

```bash
git clone https://github.com/zsh-users/zsh-autosuggestions ${ZSH_CUSTOM:-~/.oh-my-zsh/custom}/plugins/zsh-autosuggestions # 安装自动补全插件
git clone https://github.com/zsh-users/zsh-syntax-highlighting.git ${ZSH_CUSTOM:-~/.oh-my-zsh/custom}/plugins/zsh-syntax-highlighting # 安装语法高亮插件
```

然后在 `~/.zshrc` 的 `plugins=(...)` 里加上对应插件名。

#### `fzf`

`fzf` 是命令行模糊搜索工具，可以帮你更快找文件、找历史命令。

```bash
conda install -c conda-forge fzf # 安装 fzf 模糊搜索工具
```

配置思路通常是：

```bash
plugins=(git extract zsh-autosuggestions zsh-syntax-highlighting fzf) # 启用常用插件
FZF_BASE=/your/env/path/share/fzf/shell # 指定 fzf 安装路径
```

保存后重新登录或重新加载配置。

#### `extract`

这是一个解压相关的小工具插件。不是必须，但对于经常处理压缩文件的人很省事。

这一节的核心不是“美化终端”，而是减少重复输入、降低出错率、提高搜索效率。

## 常见坑

### 1. 分不清本地和服务器

很多“文件不见了”“路径不对”的问题，其实只是你搞混了自己现在到底在本地还是服务器。

### 2. 不看当前目录就执行删除命令

这个坑非常危险。删之前先：

```bash
pwd # 先确认当前目录
ls  # 再确认当前目录下的文件
```

### 3. 不开 `tmux` 就直接跑长任务

这是新人很值得尽早避开的一个坑。

### 4. 用 `cat` 打开超大文件

大文件优先用：

```bash
less file         # 翻页查看大文件
tail -n 50 file   # 看最后 50 行
grep keyword file # 搜索关键词
```

### 5. 不理解 `>` 和 `>>` 的区别

一个是覆盖，一个是追加。这个没搞清楚，日志很容易被你自己覆盖掉。

如果你现在刚入组，我建议你先把下面这些事做到能独立完成：

1. 登录服务器并配置一个 `ssh` 别名

如果这几件事都能做到，后面的很多内容就不会再总被环境问题卡住。
