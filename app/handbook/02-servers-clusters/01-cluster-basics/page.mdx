# 2.1 集群基础与使用边界

很多新手第一次用集群时，会把它想成一台“更强的服务器”。这个理解只对了一半。

集群真正重要的地方，不只是算力更强，而是它是多人共享的计算资源。你通常先登录到 `login node`，在那里改文件、整理目录、提交任务；真正的大计算会被调度系统送到 `compute node` 上运行。也就是说，登录节点更像是一个入口和工作台，计算节点才是真正干重活的地方。

所以使用集群时，最核心的动作不是“直接运行程序”，而是“提交一个作业”。这个作业会告诉系统：我要跑什么程序、需要多少核、最多跑多久、输出日志写到哪里。下面先用 `Slurm` 讲清楚这件事，再扩展到 `PBS` 和 `LSF`。

## 1. 先用 Slurm 理解提交作业

`Slurm` 是现在很常见的集群调度系统。如果一台机器使用 Slurm，你通常不会在登录节点上直接运行：

```bash
vasp_std
```

而是写一个作业脚本，比如 `run.slurm`：

```bash
#!/bin/bash
#SBATCH -J test-job
#SBATCH -p cpu
#SBATCH -N 1
#SBATCH -n 16
#SBATCH --time=12:00:00
#SBATCH -o slurm-%j.out

cd $SLURM_SUBMIT_DIR
module load vasp
srun vasp_std
```

这个脚本可以先粗略理解成两部分。前面那些 `#SBATCH` 行是在向集群申请资源，后面的普通 shell 命令才是真正要执行的计算流程。

几个最常见的参数先记住这些就够了：

- `#SBATCH -J`：作业名
- `#SBATCH -p`：分区或队列
- `#SBATCH -N`：节点数
- `#SBATCH -n`：总核数或总任务数
- `#SBATCH --time`：最长运行时间
- `#SBATCH -o`：标准输出日志文件

真正提交时，在登录节点运行：

```bash
sbatch run.slurm
```

提交成功后，Slurm 通常会返回一个 job id：

```text
Submitted batch job 123456
```

这个 `123456` 很重要。之后查任务、取消任务、看日志，都会用到它。

## 2. 查看、取消和看日志

提交之后，第一件事通常是看队列：

```bash
squeue -u $USER
```

如果任务显示 `PD`，说明还在排队；如果显示 `R`，说明已经在运行。第一次用集群时，不需要马上理解所有状态码，先知道这两个就已经够用了。

常见状态可以这样看：

- `PD`：pending，排队中
- `R`：running，运行中
- `CG`：completing，快结束了
- `CD`：completed，正常完成
- `F`：failed，失败

如果发现脚本写错了，或者资源申请不合理，可以取消任务：

```bash
scancel 123456
```

如果脚本里写了：

```bash
#SBATCH -o slurm-%j.out
```

那么日志文件通常会叫：

```text
slurm-123456.out
```

可以用下面的命令实时看输出：

```bash
tail -f slurm-123456.out
```

所以一个最小 Slurm 工作流其实很短：

```bash
sbatch run.slurm
squeue -u $USER
tail -f slurm-123456.out
scancel 123456   # 如果需要取消
```

你真正要先建立的直觉是：登录节点负责提交和管理，计算节点负责执行，调度系统负责把你的作业排到合适的资源上。

## 3. 为什么不能直接在登录节点跑

登录节点通常是很多人一起使用的入口。如果大家都直接在登录节点上跑长时间大任务，机器很快就会卡住，别人连登录、传文件、查结果都可能受影响。

所以一般规则是：

- 改文件、传文件、查目录、看日志：可以在登录节点做
- 编译小程序、测试很短的命令：通常可以在登录节点做，但要克制
- 正式计算、大规模并行任务、长时间程序：通过调度系统提交

如果不确定一件事该不该直接跑，先问组里同学或管理员。集群是共享资源，保守一点通常更安全。

## 4. PBS / qsub 怎么对应

有些集群不用 Slurm，而是用 `PBS` 或 `Torque`。这时你会看到的命令通常是：

- `qsub`：提交任务
- `qstat`：查看任务
- `qdel`：删除任务

一个最小 PBS 脚本大概长这样：

```bash
#!/bin/bash
#PBS -N test-job
#PBS -q batch
#PBS -l nodes=1:ppn=16
#PBS -l walltime=12:00:00
#PBS -o pbs.out

cd $PBS_O_WORKDIR
module load vasp
mpirun vasp_std
```

提交任务：

```bash
qsub run.pbs
```

查看自己的任务：

```bash
qstat -u $USER
```

删除任务：

```bash
qdel 123456
```

你会发现 PBS 和 Slurm 的思路是一样的：写脚本，申请资源，提交作业，查看队列，必要时删除任务。区别主要是命令名字和脚本头部的写法不同。

## 5. LSF 怎么对应

还有一些集群会用 `LSF`。它常见的三个命令是：

- `bsub`：提交任务
- `bjobs`：查看任务
- `bkill`：删除任务

LSF 可以写脚本，也可以直接在命令行里提交一个简单任务。比如：

```bash
bsub -J test-job -n 16 -q normal -W 12:00 -o lsf.%J.out "mpirun vasp_std"
```

查看任务：

```bash
bjobs
```

删除任务：

```bash
bkill 123456
```

如果任务比较复杂，仍然建议写成脚本，这样以后更容易复现，也更容易检查自己到底申请了什么资源。

## 6. 三种系统的最小对照

如果把三套系统放在一起看，最小对应关系是：

| 功能 | Slurm | PBS/Torque | LSF |
| --- | --- | --- | --- |
| 提交任务 | `sbatch` | `qsub` | `bsub` |
| 查看任务 | `squeue` | `qstat` | `bjobs` |
| 删除任务 | `scancel` | `qdel` | `bkill` |
| 作业脚本头 | `#SBATCH` | `#PBS` | `#BSUB` |

所以你不需要把它们看成三套完全不同的东西。先用 Slurm 建立直觉，再遇到 PBS 或 LSF 时，只要把“提交、查看、删除、日志”这几个动作对应过去就行。

## 7. 新手最容易踩的坑

最常见的不是高级错误，而是下面这些非常基础的误区：

- 在登录节点直接跑长时间大任务
- 不知道集群到底用的是 `Slurm`、`LSF` 还是 `PBS`
- 把任务目录和结果长期全堆在 `home`
- 任务出问题时不先看队列状态和日志
- 申请了很多核，但程序实际只用了一个核
- 日志文件没看，就直接判断“任务挂了”

所以第一次上手时，一个非常实用的顺序是：

1. 先问清楚集群用的是什么调度系统
2. 找一个组里已经能跑通的最小作业脚本
3. 先照着改任务名、核数、时间和程序名
4. 提交后先查队列，再看日志

等这套流程跑通以后，再去细看分区、节点类型、内存、GPU、数组任务和依赖任务。对刚开始的人来说，先把一个最小作业稳定提交、查到、看懂日志，比一开始背完所有参数更重要。
