# 2.1 集群基础与使用边界

很多新手第一次用集群时，会把它想成一台“更强的服务器”。

但集群真正重要的地方，不只是算力更强，而是：

- 机器是多人共享的
- 计算资源需要排队和分配
- 你通常不能想跑什么就直接在登录节点跑什么

所以这篇想先帮你建立一个最小工作图景：

1. 集群里不同节点分别负责什么
2. 为什么要通过调度系统提交任务
3. `Slurm`、`LSF`、`PBS/qsub` 这些名字到底在说什么

## 1. login node 和 compute node

最先要分清的就是这两个角色：

- `login node`：登录、改文件、传文件、提交任务、查日志
- `compute node`：真正执行计算程序

所以一般不应该在登录节点上长期直接跑大任务。

更常见的正确姿势是：

```bash
ssh alias
cd ~/work/project-a
sbatch run.slurm
```

也就是：

- 登录节点负责“组织任务”
- 计算节点负责“执行任务”

## 2. `home`、`work`、`scratch`

不同机器命名不完全一样，但常见思路是：

- `home`：个人配置、小文件、脚本，容量通常不大
- `work`：常用项目目录
- `scratch`：大规模临时数据，读写快，但可能会定期清理

所以不要默认把所有重要结果长期堆在同一个地方。

一个很常见的坏习惯是：

- 把大计算和大输出全放在 `home`

一个更合理的习惯通常是：

- 配置和小脚本放 `home`
- 任务目录放 `work` 或 `scratch`
- 重要结果及时整理和备份

## 3. 为什么需要调度系统

多人共用算力时，系统需要回答这些问题：

- 谁先跑
- 占多少核
- 占多少内存
- 最多跑多久
- 跑到哪台机器上

这就是调度系统在做的事。

你平时会看到的几个名字：

- `Slurm`
- `LSF`
- `PBS` / `Torque`

它们本质上都在做同一类事，只是命令名字不一样。

## 4. 先用 Slurm 讲最小流程

如果你是第一次接触调度系统，最适合先用 `Slurm` 建立直觉，因为它现在很常见，而且命令相对统一。

### 4.1 一个最小作业脚本

先看一个最小例子：

```bash
#!/bin/bash
#SBATCH -J test-job
#SBATCH -p cpu
#SBATCH -N 1
#SBATCH -n 16
#SBATCH --time=12:00:00
#SBATCH -o slurm-%j.out

cd $SLURM_SUBMIT_DIR
module load vasp
srun vasp_std
```

你现在先不要追求把每个参数都背下来，只要先建立这些直觉：

- `#SBATCH -J`：任务名
- `#SBATCH -p`：分区 / 队列
- `#SBATCH -N`：节点数
- `#SBATCH -n`：任务总核数
- `#SBATCH --time`：最长运行时间
- `#SBATCH -o`：标准输出日志文件

脚本正文里最常见的几步通常是：

1. 回到提交目录
2. 加载环境
3. 启动程序

### 4.2 提交任务

```bash
sbatch run.slurm
```

提交后通常会返回一个 job id，例如：

```text
Submitted batch job 123456
```

这个 `123456` 后面会很有用，因为查状态、取消任务都常靠它。

### 4.3 查看队列

最常用的是：

```bash
squeue -u $USER
```

你可以先把它理解成：

- 看我现在有哪些任务在排队或运行

如果任务还没开始，常见原因通常是：

- 分区资源不够
- 你申请的资源太多
- 前面还有别人的任务

### 4.4 查看任务状态

在 `Slurm` 里，最常见的几个状态你先记住就够了：

- `PD`：pending，排队中
- `R`：running，运行中
- `CG`：completing，快结束了
- `CD`：completed，正常完成
- `F`：failed，失败

### 4.5 取消任务

```bash
scancel 123456
```

也可以按名字或用户取消，但第一次先按 job id 来做最稳。

### 4.6 看日志

如果脚本里写了：

```bash
#SBATCH -o slurm-%j.out
```

那么日志文件通常会叫：

```text
slurm-123456.out
```

最常见的看法是：

```bash
tail -f slurm-123456.out
```

### 4.7 一个最小 Slurm 工作流

把它压缩成最短流程，通常就是：

```bash
sbatch run.slurm
squeue -u $USER
tail -f slurm-123456.out
scancel 123456   # 如果需要取消
```

如果你现在还没形成队列系统直觉，先把这四步记住就够了。

## 5. LSF 怎么理解

有些老集群或学校平台用的是 `LSF`。

它和 `Slurm` 做的是同一类事，只是命令换了名字。

你最常见到的会是：

- `bsub`：提交任务
- `bjobs`：查看任务
- `bkill`：取消任务

一个最小例子可以先这样看：

```bash
bsub -J test-job -n 16 -q normal -W 12:00 -o lsf.%J.out "mpirun vasp_std"
```

你可以先把它粗略对应到 `Slurm`：

- `bsub` 类似 `sbatch`
- `bjobs` 类似 `squeue`
- `bkill` 类似 `scancel`

所以第一次遇到 `LSF` 时，不要慌，本质上你还是在做同一件事：

- 申请资源
- 提交任务
- 查状态
- 看日志

## 6. PBS / qsub 怎么理解

还有一些机器会用 `PBS` 或 `Torque`，最显眼的命令一般是：

- `qsub`：提交任务
- `qstat`：查看任务
- `qdel`：删除任务

一个很典型的脚本长这样：

```bash
#!/bin/bash
#PBS -N test-job
#PBS -q batch
#PBS -l nodes=1:ppn=16
#PBS -l walltime=12:00:00
#PBS -o pbs.out

cd $PBS_O_WORKDIR
module load vasp
mpirun vasp_std
```

提交：

```bash
qsub run.pbs
```

查看：

```bash
qstat -u $USER
```

删除：

```bash
qdel 123456
```

## 7. 把三种系统先对照起来

如果你刚开始最怕的是“命令名字太多记不住”，那就先看这个最小对照：

| 功能 | Slurm | LSF | PBS/Torque |
| --- | --- | --- | --- |
| 提交任务 | `sbatch` | `bsub` | `qsub` |
| 查看任务 | `squeue` | `bjobs` | `qstat` |
| 删除任务 | `scancel` | `bkill` | `qdel` |
| 交互式运行 | `srun` / `salloc` | `bsub -Is` | `qsub -I` |

所以你真正要掌握的不是三套完全不同的逻辑，而是同一套逻辑的三种方言。

## 8. 新手最容易踩的坑

最常见的不是高级错误，而是下面这些非常基础的误区：

- 在登录节点直接跑长时间大任务
- 脚本里申请了 16 核，程序却只用 1 核
- 日志文件都没看，就直接说“任务挂了”
- 不知道自己机器到底用的是 `Slurm`、`LSF` 还是 `PBS`
- 把任务目录和结果长期全堆在 `home`

所以第一次上手时，一个非常实用的顺序是：

1. 先问清楚集群用的是什么调度系统
2. 找一个组里已经能跑通的最小作业脚本
3. 先照着改任务名、核数、时间和程序名
4. 提交后先查队列，再看日志
