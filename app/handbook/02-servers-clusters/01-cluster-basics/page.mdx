# 2.1 集群基础与使用边界

很多新手第一次用集群时，会把它想成一台“更强的服务器”。

但集群真正重要的地方，不只是算力更强，而是：

- 机器是多人共享的
- 资源需要统一调度
- 你通常不能在登录节点随意直接跑大任务

所以这一篇只想帮你先建立一个最小图景：

1. 集群里不同节点各自负责什么
2. 常见文件系统怎么分工
3. `Slurm`、`LSF`、`PBS/qsub` 这些名字到底是什么

## 1. login node 和 compute node

最先要分清的就是这两个角色：

- `login node`：登录、改文件、传文件、提交任务、查日志
- `compute node`：真正执行计算程序

所以一般不应该在登录节点上长期直接跑大任务。

更常见的正确姿势是：

```bash
ssh alias
cd ~/work/project-a
sbatch run.slurm
```

也就是：

- 登录节点负责“组织任务”
- 计算节点负责“执行任务”

## 2. `home`、`work`、`scratch`

不同机器命名不完全一样，但常见思路是：

- `home`：个人配置、小文件、脚本，容量通常不大
- `work`：常用项目目录
- `scratch`：大规模临时数据，读写快，但可能会定期清理

所以不要默认把所有重要结果长期堆在同一个地方。

一个很常见的坏习惯是：

- 把大计算和大输出全放在 `home`

一个更合理的习惯通常是：

- 配置和小脚本放 `home`
- 任务目录放 `work` 或 `scratch`
- 重要结果及时整理和备份

## 3. 为什么需要调度系统

多人共用算力时，系统需要回答这些问题：

- 谁先跑
- 占多少核
- 占多少内存
- 最多跑多久
- 跑到哪台机器上

这就是调度系统在做的事。

你平时会看到的几个名字：

- `Slurm`
- `LSF`
- `PBS` / `Torque`

它们本质上都在做同一类事，只是命令名字不一样。

你真正要掌握的不是三套完全不同的逻辑，而是同一套逻辑的三种方言。

## 4. 三种系统先怎么粗略理解

如果你只想先知道它们分别大概长什么样，可以先这样记：

- `Slurm`：`sbatch / squeue / scancel`
- `LSF`：`bsub / bjobs / bkill`
- `PBS`：`qsub / qstat / qdel`

也就是说，它们都在做这三件事：

1. 提交任务
2. 查看任务
3. 删除任务

更具体的配套软件和结构处理工具，放到第三章的 [3.2 常用结构处理与可视化工具](/handbook/03-dft/02-structure-visualization-tools) 里讲。

## 5. 新手最容易踩的坑

最常见的不是高级错误，而是下面这些非常基础的误区：

- 在登录节点直接跑长时间大任务
- 不知道集群到底用的是 `Slurm`、`LSF` 还是 `PBS`
- 把任务目录和结果长期全堆在 `home`
- 任务出问题时不先看队列状态和日志

所以第一次上手时，一个非常实用的顺序是：

1. 先问清楚集群用的是什么调度系统
2. 找一个组里已经能跑通的最小作业脚本
3. 先照着改任务名、核数、时间和程序名
4. 提交后先查队列，再看日志

## 6. 一个保守但很实用的原则

如果你暂时不确定一件事该不该做，先遵循这个规则：

- 改文件、传文件、查日志：登录节点
- 真正的大计算：通过调度系统提交
- 重要结果：及时整理和备份
